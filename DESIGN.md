## 1. Краткое описание продукта

Сервис позволяет любой команде создать публичную **страницу статуса** своего продукта:
перечислить сервисы (компоненты), вручную управлять их состоянием, публиковать инциденты и
плановые работы, а также давать клиентам подписываться на обновления (email / RSS / Telegram /
календарь / webhook).

**Ключевая философия — manual-first.** В отличие от мониторинговых сервисов (PingMap, Statuser),
где статус компонента меняется автоматически по результатам TCP/HTTP/gRPC-проверок, здесь
оператор управляет статусом и инцидентами **вручную**. Автоматизация (приём алертов от Grafana /
Prometheus / PagerDuty / собственного мониторинга через API и webhook) — опциональный слой
поверх ручного ядра, а не его замена.

Это прямой функциональный аналог Статусмейт и Statuspal: ядро — коммуникация об инцидентах,
а не probe-based мониторинг.

**[РЕШЕНО] Нейминг — HealthPage / healthpage.ru.** Имя оставлено владельцем. Известный риск:
на русском рынке «health» сильнее ассоциируется с медициной, чем с аптаймом сервисов (у
конкурентов в названии есть прямой сигнал «status/ping»); компенсируется маркетингом и сильным
продуктом. Для агента: использовать «HealthPage» как имя продукта и `healthpage.ru` как
основной домен.

## 2. Целевые пользователи и роли

| Роль | Описание | Доступ |
|------|----------|--------|
| **Owner** | Создатель аккаунта/страницы, биллинг | Полный |
| **Admin** | Управляет настройками страницы, командой | Всё, кроме биллинга/удаления страницы |
| **Editor** | Создаёт и ведёт инциденты, работы, меняет статусы компонентов | CRUD инцидентов/работ/компонентов |
| **Viewer (internal)** | Только чтение внутренней панели | Read-only админка |
| **Subscriber** | Внешний клиент, подписан на обновления | Только публичная часть |
| **Visitor** | Анонимный посетитель публичной страницы | Только публичная часть |

**[ТРЕБОВАНИЕ]** Уровни доступа (роли) настраиваются на уровне страницы. Один пользователь
может иметь разные роли на разных страницах.

## 3. Функциональные требования

### 3.1 Управление страницами статуса
- **[ТРЕБОВАНИЕ]** Один аккаунт может владеть несколькими страницами статуса.
- Каждая страница имеет: название, описание, slug (поддомен `slug.example.ru`), часовой пояс,
  язык по умолчанию, тему оформления.
- Публичность страницы: `public` | `private` (доступ по паролю или по списку email).
- **[ТРЕБОВАНИЕ]** Приватная страница не индексируется (`noindex`) и не доступна без авторизации.

### 3.2 Компоненты (сервисы)
Компонент = единица сервиса, чьё состояние показывается клиенту (например: Web-сайт, API, БД,
Серверы, Сеть).
- CRUD компонентов; сортировка (drag-and-drop, поле `position`).
- **Группы и подкомпоненты** — компоненты образуют дерево через поле `parent_id`
  (как у Статусмейта: компонент может иметь родителя). Группа/родитель имеет агрегированный
  статус (худший из дочерних).
- **[РЕШЕНО]** Поддерживаем **подкомпоненты** (древовидная вложенность через `parent`), как у
  Статусмейта — не ограничиваемся двумя уровнями. Модель уже это позволяет (`parent_id`);
  UI должен уметь отображать и редактировать вложенность. Разумное ограничение глубины
  (например, 3–4 уровня) — на усмотрение реализации, чтобы не сломать вёрстку, но архитектурно
  дерево произвольное.
- **Приватные компоненты:** не отображаются на публичной странице; оповещения о них получают
  только подписчики именно этих компонентов.
- Опция «показывать график uptime» для компонента (вкл/выкл).
- Опция «не показывать состояние» (компонент как информационная метка).

#### Статусы компонента (enum, нормативно)
```
operational            # Работает
degraded_performance   # Снижение производительности
partial_outage         # Частичная недоступность
major_outage           # Полная недоступность
under_maintenance      # На обслуживании
```
Состояние компонента меняется: (а) вручную оператором, (б) автоматически — как производное
от активного инцидента/работы, затрагивающего компонент (см. 3.3).

### 3.3 Инциденты
Инцидент = публичный отчёт о сбое с хронологией обновлений.
- CRUD инцидентов.
- Поля: заголовок, текущий статус инцидента, уровень влияния (impact), список затронутых
  компонентов с их состоянием в рамках инцидента, лента обновлений (updates), время начала.
- **Промежуточные статусы (нормативно):**
```
investigating   # Идёт расследование
identified      # Причина обнаружена
monitoring      # Наблюдаем за исправлением
resolved        # Устранено
```
- **Уровни влияния (impact, нормативно):**
```
none | minor | major | critical
```
- **Лента обновлений:** каждое обновление — отдельная запись (`IncidentUpdate`) со статусом,
  текстом (с поддержкой переносов строк и Markdown), временной меткой и флагом «уведомить
  подписчиков». Это даёт честную хронологию «расследуем → нашли → исправили».
- **Постмортем:** после `resolved` можно прикрепить пост-мортем (отдельное поле/обновление).
- **[ТРЕБОВАНИЕ]** При создании/обновлении инцидента состояние затронутых компонентов
  обновляется автоматически согласно тому, что указал оператор в отчёте; при `resolved`
  компоненты возвращаются в `operational`, если оператор не указал иное.
- **Шаблоны инцидентов** (быстрые шаблоны): преднастроенные заготовки для типовых ситуаций.
- **Фильтрация** инцидентов по компонентам и по уровню влияния.

### 3.4 Плановые работы (Maintenance)
- CRUD запланированных работ.
- Поля: заголовок, описание, `scheduled_start`, `scheduled_end`, затронутые компоненты,
  лента обновлений.
- **Статусы работ (нормативно):**
```
scheduled     # Запланировано
in_progress   # Выполняется
completed     # Завершено
```
- Анонс заранее: уведомление подписчиков при создании и при смене статуса.
- На время `in_progress` затронутые компоненты получают `under_maintenance`.
- Архив выполненных работ с хронологией.

### 3.5 Подписки и уведомления
- **Каналы подписки:**
  - **Email** (подтверждение double opt-in).
  - **RSS / Atom** (публичный фид страницы).
  - **Календарь** (iCal-фид плановых работ — подписка на календарь).
  - **Telegram** (бот; подписка на страницу или на отдельные компоненты).
  - **MAX** (российский мессенджер; бот через MAX Bot API). **Первоклассный канал** для
    выхода на рынок РФ — реализуется наравне с Telegram, не опционально. Детали в разделе 4.1.
  - **Slack** (подписка канала Slack клиента на обновления). **Обязательный канал для запуска.**
    Подписчик добавляет интеграцию через OAuth («Add to Slack»), выбирает канал, и туда
    приходят уведомления об инцидентах/работах — так же, как email/Telegram/MAX. Это
    подписка-канал, а не ручной webhook (см. отличие в разделе 4.4 и примечание в разделе 4).
- **[ТРЕБОВАНИЕ]** Подписчик может подписаться на всю страницу **или** на конкретные
  компоненты (в т.ч. приватные, если у него есть доступ).
- **[ТРЕБОВАНИЕ]** Нет лимита на количество подписчиков (продуктовая позиция Статусмейт).
- Письма: уведомление о новом инциденте, о каждом обновлении (если помечено «уведомить»),
  о начале/окончании плановых работ, о смене статуса компонента (опционально).
- **Кастомный SMTP** — страница может указать собственный SMTP-сервер.
- **Собственный адрес отправителя** (custom From) — премиум.
- **Управление отпиской** — ссылка отписки в каждом письме, страница управления подпиской.

### 3.6 Кастомизация и брендирование
- Загрузка логотипа и favicon.
- Выбор макета (layout) и цветовой схемы (фирменный стиль).
- Тёмная тема.
- Скрытие плашки «Работает на …» (white-label) — премиум.
- Настройка формата времени и часового пояса.
- Перенаправление на URL (страница может редиректить на внешний адрес).
- **Собственный домен** (CNAME) с автоматическим выпуском TLS-сертификата (премиум).

### 3.7 Прочее
- **Виджет статуса** — встраиваемый бейдж/баннер (JS-сниппет или SVG-бейдж) для сайта клиента.
- **Страница релизов / changelog** — лента анонсов продукта (отдельно от инцидентов).
- **История инцидентов** — публичный архив с пагинацией.
- **Расчёт uptime** — % доступности по компоненту за 90 дней (на основе истории состояний/инцидентов).

## 4. Интеграции (опциональный слой автоматизации)

Ядро ручное, но предусмотрены входящие интеграции для авто-создания инцидентов:

| Источник | Механизм | Что делает |
|----------|----------|------------|
| **Grafana** | Webhook (alert) | Создаёт/обновляет инцидент по алерту |
| **Prometheus (Alertmanager)** | Webhook | Создаёт инцидент при срабатывании правила |
| **PagerDuty** | Webhook/Events API | Создаёт инцидент при PD-инциденте |
| **Mattermost / Slack** | Исходящий webhook | Постит обновления в канал по произвольному webhook URL (технический путь). Для Slack основной способ — канал подписки через OAuth, см. 4.4 |
| **Telegram-боты** | Bot API | Уведомляет подписчиков; команды управления |
| **Внешний мониторинг** | REST API (раздел 7) | Любой источник через токен |

**[ТРЕБОВАНИЕ]** Каждый входящий webhook аутентифицируется (подписанный секрет/HMAC или токен)
и маппится на конкретную страницу и набор компонентов. Маппинг настраивается оператором.

### 4.1 Интеграция с мессенджером MAX (исходящие уведомления)

MAX — российский мессенджер (VK). Используется как канал доставки уведомлений подписчикам
(аналогично Telegram). Сервис выступает в роли **бота MAX**, рассылающего обновления статуса.

**Технические факты (актуально на 2026, подлежат проверке перед реализацией):**
- REST API на домене `platform-api.max.ru` (идёт миграция; в части источников упоминается
  `platform-api2.max.ru` — **[ТРЕБОВАНИЕ]** перед разработкой свериться с `https://dev.max.ru/docs-api`).
- Лимит API — порядка 30 запросов/сек. Это влияет на throughput воркера рассылки — см. ниже.
- Аутентификация: токен в заголовке `Authorization: <token>`. Передача токена через
  query-параметры **не поддерживается**.
- Получение событий: **Webhook** (для production; требует HTTPS и доверенный сертификат,
  в т.ч. сертификат Минцифры) либо Long Polling (только для разработки).
- Получение токена бота — через `@MasterBot` внутри мессенджера / консоль `dev.max.ru`.
- Есть официальные клиентские библиотеки (TypeScript, **Go**, Python) — для Go-бэкенда
  использовать официальную Go-библиотеку.

**[ТРЕБОВАНИЕ] Регуляторное ограничение (статус юрлица для публикации бота):**
Публикация бота в MAX требует верифицированного профиля на «Платформе MAX для партнёров»
и прохождения модерации. **С 15 июня 2026 доступ открыт для самозанятых** (резидентов РФ) —
наравне с юрлицами и ИП. Самозанятый верифицируется **только через Госуслуги**. Это
соответствует выбранному статусу владельца (самозанятость) — отдельное юрлицо для MAX-бота
оформлять не нужно.
Нюансы:
- Цифровой ID на платформе MAX доступен только юрлицам и ИП; для бота уведомлений он не требуется,
  так что на нашу задачу не влияет.
- Бот всё равно проходит модерацию; в нём обязательны пользовательское соглашение, политика
  конфиденциальности, реквизиты правообладателя и канал поддержки.
- Ник бизнес-профиля самозанятого формируется по шаблону `se(orgid)_...`.
**Вывод для плана:** организационный блокер снят — верификация самозанятого через Госуслуги
плюс модерация бота. Запускать заранее, но это уже не зависит от создания юрлица.

**Реализация в архитектуре:**
- Отдельный воркер `worker-max` — потребитель очереди `q.max` (RabbitMQ), доставляет
  уведомления через MAX Bot API. Симметрично `worker-telegram`.
- Throughput-лимит 30 rps учитывается в воркере (rate limiter перед вызовами API);
  при пиковых рассылках сообщения остаются в `q.max` и обрабатываются с нужной скоростью.
- Подписка: пользователь стартует бота MAX → бот получает `bot_started`/`message_created` →
  сервис создаёт `Subscriber{channel=max, address=chat_id}`. Привязка к странице — через
  deep-link/команду с кодом страницы.
- Тот же `chat_id` хранится в `Subscriber.address`, как и для Telegram.

### 4.2 Платежи и биллинг (приём оплаты за Premium)

Premium-тариф (раздел 10) требует приёма онлайн-платежей. Целевой рынок — РФ.
**Закреплённый провайдер: ЮKassa** (ООО НКО «ЮMoney», экосистема Сбера) — один из самых
функциональных эквайрингов в РФ: широкий охват методов (карты, СБП, ЮMoney, SberPay/T-Pay),
рекуррентные списания, простые интеграции, отсутствие абонентской платы (комиссия только за
успешные платежи).

**[ТРЕБОВАНИЕ] Абстракция провайдера сохраняется.** Несмотря на выбор ЮKassa, платёжная
логика инкапсулируется за интерфейсом `PaymentProvider` (Go). Причина — см. ниже про
ограничения самозанятости: вероятна смена статуса/провайдера по мере роста, и доменный код
не должен от этого зависеть.

**Модель оплаты:** подписка с рекуррентными (автоматическими) списаниями — помесячно/ежегодно.
Первая оплата сохраняет платёжный токен карты у провайдера, последующие списания идут
автоматически.

> **[РЕШЕНО для MVP] Статус «самозанятый» + ЮKassa — осознанный выбор на старте.**
> Владелец оформлен как **самозанятый** (НПД). Это сознательное решение для MVP: цель запуска —
> **проверить спрос** (готов ли рынок платить за Premium), а не масштабироваться. Удорожание
> (ИП, онлайн-касса/ОФД, снятие лимитов) намеренно НЕ закладывается на старте.
> **[ТРЕБОВАНИЕ для агента]** Не усложнять биллинг под B2B и масштаб сверх описанного ниже.
> Реализовать минимальный рабочий приём оплаты в рамках ограничений самозанятости.
>
> **Известные границы MVP (это рамки, а не нерешённые вопросы):**
> - Плательщик — **физлицо** (личная карта, self-service, рекуррент). Оплата от юрлиц/ИП через
>   эквайринг не поддерживается — для MVP это приемлемо.
> - Годовой потолок НПД — 2,4 млн ₽; месячный лимит ЮKassa по умолчанию ~500 000 ₽
>   (повышается через менеджера). Для проверки гипотезы этого достаточно.
> - Рекурренты включаются по согласованию с менеджером ЮKassa (скриншот механизма отвязки
>   карты + оценка оборота).
> - Фискализация: онлайн-касса/ОФД **не нужна** — чеки через «Мой налог», ЮKassa передаёт в ФНС.
>
> **Путь апгрейда (на будущее, НЕ реализуется сейчас).** Если гипотеза подтвердится и упрётся
> в потолок/в спрос со стороны компаний — переход на ИП. Чтобы этот переход был дешёвым,
> единственное, что закладывается архитектурно сейчас, — абстракция `PaymentProvider` (Go)
> и хранение `Payment.receipt_id` независимо от способа фискализации. Больше ничего «на вырост»
> не делать.

**[ТРЕБОВАНИЕ] Фискализация (54-ФЗ) для MVP — режим НПД:**
- Самозанятый: отдельная онлайн-касса/ОФД **не нужна**. Чеки формируются в режиме НПД;
  ЮKassa автоматически передаёт сведения в ФНС и формирует чек через «Мой налог».
- В модели `Payment.receipt_id` хранит ссылку на чек (поле остаётся пригодным и для ОФД,
  если в будущем будет апгрейд на ИП — но сейчас ОФД не реализуется).

**Поток оплаты (провайдер-агностичный):**
1. Клиент выбирает Premium → создаётся `Subscription{status=pending}` и платёж в ЮKassa.
2. Клиент оплачивает (редирект/виджет ЮKassa); сохраняется платёжный токен для автосписаний.
3. ЮKassa шлёт **webhook** о результате → сервис верифицирует подпись, переводит `Subscription`
   в `active`, включает Premium feature-flags.
4. По окончании периода `worker-billing` инициирует рекуррентное списание по токену.
5. При неуспехе — ретраи по графику (dunning), уведомление клиенту; при исчерпании —
   `past_due` → `canceled`, откат на Free.
6. Чек формируется на каждое успешное списание через «Мой налог» (режим НПД).

**[ТРЕБОВАНИЕ] Безопасность и корректность:**
- Данные карт **не хранятся** на нашей стороне — только платёжный токен ЮKassa (PCI DSS на провайдере).
- Идемпотентность платёжных операций (idempotency-key) и защита от повторной оплаты.
- Верификация подписи входящих webhook'ов ЮKassa.
- Журнал всех платёжных событий (audit log) для сверки и возвратов.
- Нужна публичная **оферта** с условиями подключения/отключения автоплатежа (требование ЮKassa).

### 4.3 Миграция клиентов (мультиисточник)

Импортёр — **инструмент go-to-market**: снижает барьер перехода для клиентов, уже
использующих другую status-page-платформу (тот же ход делает Статусмейт, предлагая «Перейти
с StatusPal/Instatus»). Цель — за один шаг перенести структуру сервисов, историю
инцидентов/работ и подписчиков.

**[ТРЕБОВАНИЕ] Архитектура — паттерн «адаптер».** Все источники реализуют единый интерфейс
`Importer` (Go); доменный код импорта не зависит от конкретного источника. Добавление нового
источника = новый адаптер, без изменения ядра.

```go
type Importer interface {
    // Проверка доступа и предпросмотр объёма (что будет перенесено)
    Probe(ctx, creds) (ImportPreview, error)
    // Вытягивание нормализованных сущностей источника
    FetchComponents(ctx, creds) ([]SourceComponent, error)
    FetchIncidents(ctx, creds)  ([]SourceIncident, error)
    FetchMaintenances(ctx, creds) ([]SourceMaintenance, error)
    FetchSubscribers(ctx, creds) ([]SourceSubscriber, error)
}
```
Каждый адаптер маппит свои данные в общие промежуточные структуры (`Source*`), а общий слой
импорта пишет их в нашу модель через `ExternalIdMap` (идемпотентность). Источник фиксируется
в `ImportJob.source` (enum `statuspal | statuspage | instatus | statusmate`).

**Общие требования ко всем источникам:**
- Импорт асинхронный (`worker-import`, очередь `q.import`), пагинируемый, с прогрессом и
  итоговым отчётом (перенесено / пропущено / ошибки).
- **Идемпотентность** через `ExternalIdMap`; режимы `skip` (пропускать существующие) и
  `update` (обновлять).
- Импорт исторических инцидентов/работ **не рассылает уведомления** (флаг `notify=false` на
  всех создаваемых сущностях) — иначе подписчики получат спам о старых событиях.
- SMS-подписки источников **не импортируются** (у нас нет SMS-канала, см. 3.5) — идут в отчёт
  как пропущенные.
- Учётные данные источника (`api_key`) не хранятся дольше задачи.
- **[ВЕРНУТЬСЯ ПЕРЕД ЗАПУСКОМ ФУНКЦИОНАЛА МИГРАЦИИ] 152-ФЗ / 38-ФЗ — согласие подписчиков.**
  Это организационно-юридический вопрос, который **обязательно** пересмотреть до включения
  импорта подписчиков в проде (но не блокирует разработку остального).
  Итог проведённого исследования (нормативная рекомендация по умолчанию):
  - Для РКН не важно, как получена база (перенос, покупка, сбор) — важно **наличие согласия**.
    Согласие на обработку ПД (email) давалось исходной платформе, не нам.
  - Отраслевой best-practice при импорте чужой базы: **повторный double opt-in** — первым
    письмом предложить подтвердить подписку или отказаться. Почтовые платформы даже при
    «автоактивации» готовой базы часто требуют повторного DOI.
  - Нюанс в нашу пользу: уведомления статус-страницы об инцидентах — это **сервисные**
    (транзакционные) сообщения, а не реклама; для них предварительное согласие по 38-ФЗ
    «О рекламе» не требуется. Но согласие на обработку ПД по 152-ФЗ всё равно нужно.
  - **Решение по умолчанию для реализации:** импортированные email **НЕ** помечать как
    подтверждённые автоматически; ставить `confirmed=false` и при первом контакте отправлять
    письмо-подтверждение (opt-in). Это безопасный дефолт; владелец может пересмотреть после
    юридической консультации (например, авто-подтверждение, если получит подтверждение
    законности от юриста).
  - Конкретную публичную позицию Статусмейта по этому вопросу найти не удалось — они
    декларируют перенос подписчиков, но механику согласия не раскрывают. Поэтому ориентир —
    общий best-practice выше, а не их (неизвестная) практика.

**Приоритет источников:**
1. **StatusPal** и **Instatus** — основной go-to-market на международных клиентов (стабильные
   публичные API). Реализуются первыми.
2. **Atlassian Statuspage** — крупный источник, но низкий приоритет для рынка РФ.
3. **Статусмейт** — прямой конкурент на рынке РФ; см. предупреждение в 4.3.3.

#### 4.3.1 Источник: StatusPal (API v2)
- Базовый URL: `https://statuspal.io/api/v2` (US) или `https://statuspal.eu/api/v2` (EU) —
  **[ТРЕБОВАНИЕ]** дать выбор региона, данные изолированы по серверам.
- Авторизация: API-ключ в заголовке `Authorization`. Идентификатор страницы — `subdomain`.
- Даты — UTC ISO 8601. Rate limit: EU 300/10с, US 100/10с (троттлинг в адаптере).

| Наша сущность | Эндпоинт StatusPal | Примечание |
|---------------|--------------------|------------|
| Component / ComponentGroup | `GET /status_pages/{subdomain}/services` | «services» с иерархией parent/child (`children[]`) и флагом `private` |
| Incident (+ updates) | `GET /status_pages/{subdomain}/incidents` | Вместе с `updates[]` и `service_ids` |
| Maintenance (+ updates) | `GET /status_pages/{subdomain}/summary` | `type=scheduled`, `starts_at`/`ends_at` |
| Subscriber | `GET /status_pages/{subdomain}/subscriptions` | Email/SMS (SMS не импортируем) |
| Метаданные страницы | `GET /status_pages/{subdomain}/status` | name, time_zone, url |

Маппинг полей StatusPal:
- *Services → Component/ComponentGroup:* `service` с непустым `children[]` → `ComponentGroup` +
  дочерние `Component`; лист → `Component`. `service.private==true` → `is_private`. `id`
  сохраняем в `ExternalIdMap`.
- *Incidents → Incident+IncidentUpdate:* `title`→`title`, `starts_at`→`started_at`;
  `incident.type` → `incident_impact` (`major→major`, `minor→minor`, кастомные → по серьёзности,
  по умолчанию `minor`, **[РЕШИТЬ]** таблица); `service_ids[]`→`IncidentComponent[]`;
  `affects_uptime` учитываем в uptime. Каждый `updates[]` → `IncidentUpdate`:
  `description`/`description_html`→Markdown `body`, `posted_at`→`created_at`,
  `update.type`→`incident_status` (`investigating/identified/monitoring/resolved` напрямую,
  `issue`/`update`/прочее → `investigating` или статус предыдущего; **[РЕШИТЬ]** таблица).
  `current_status` = статус последнего обновления; при `resolved` — `resolved_at`.
  `translations` — локаль по умолчанию, остальные в i18n если включён.
- *Maintenances → Maintenance:* `title`, `starts_at→scheduled_start`, `ends_at→scheduled_end`,
  `service_ids[]→MaintenanceComponent[]`; статус по датам (прошлое→`completed`,
  сейчас→`in_progress`, будущее→`scheduled`).
- *Subscriptions → Subscriber:* email → `Subscriber{channel=email}`.

#### 4.3.2 Источник: Instatus (REST API)
- Базовый URL: `https://api.instatus.com`. Авторизация: `Authorization: Bearer <API_KEY>`
  (ключ — в `dashboard.instatus.com/developer`). Есть официальная OpenAPI-спека
  (`github.com/instatushq/openapi`).
- Эндпоинты (по сущностям): Status Pages, Components, Incidents, Incident Updates,
  Maintenances, Maintenance Updates, Subscribers.
- **[ТРЕБОВАНИЕ]** адаптер строится по официальной OpenAPI-спеке Instatus; маппинг
  аналогичен StatusPal: components (с группировкой) → наши Component/ComponentGroup,
  incidents+updates → Incident/IncidentUpdate, maintenances → Maintenance, subscribers
  (email) → Subscriber. Точные имена полей берутся из спеки на момент реализации.

#### 4.3.3 Источник: Статусмейт (API)
- Прямой конкурент на рынке РФ. Миграция технически возможна: API Статусмейта поддерживает
  чтение — подтверждён `GET https://status.statusmate.ru/api-components-list/`; токен страницы
  передаётся в заголовке `Authorization`. Логически есть list-эндпоинты и для подписчиков и
  инцидентов (разделы «Управление подписчиками»/«Управление инцидентами» в их API).
- Доступ — на уровне страницы (токен страницы), т.е. мигрирует сам владелец, имеющий доступ к
  настройкам своей страницы в Статусмейте. Это корректно с точки зрения прав.
- Поля Статусмейта близки к нашим (`uuid`, `name`, `parent`, `private`, `impact`,
  `start_date`, `histogram`), что упрощает маппинг: `impact` компонента ≈ наш
  `component_status`, `parent` → иерархия групп, `private` → `is_private`.
- **[ТРЕБОВАНИЕ] Best-effort, нестабильный источник.** API конкурента может измениться или
  закрыться без предупреждения. Адаптер Статусмейта помечается как best-effort, версия API —
  внешняя и неконтролируемая; приоритет ниже, чем у StatusPal/Instatus. Не делать его
  блокирующей зависимостью ни для какого этапа.

### 4.4 Slack как канал подписки (обязателен для запуска)

Slack — **первоклассный канал подписки** наравне с email/Telegram/MAX: клиент подписывает свой
рабочий Slack-канал на обновления статус-страницы, и туда автоматически приходят уведомления об
инцидентах и плановых работах.

**Отличие от исходящего webhook (важно, не путать).** В разделе 4 есть «исходящий webhook
Mattermost/Slack» — это **технический** путь: владелец вручную вставляет Incoming Webhook URL в
настройки, один канал на интеграцию, без UI и самообслуживания. Канал подписки 4.4 — это
**основной** путь: подписчик сам подключает Slack через OAuth, без возни с URL. Технический
webhook остаётся как legacy-вариант для тех, кому нужен произвольный endpoint.

**Механизм (Slack OAuth + Incoming Webhooks через приложение):**
- Регистрируется **Slack App** (HealthPage) с OAuth-флоу и scope `incoming-webhook`.
- Подписчик на публичной странице нажимает «Add to Slack» → проходит OAuth → выбирает рабочее
  пространство и канал → Slack возвращает webhook URL, привязанный к выбранному каналу.
- Сервис сохраняет `Subscriber{channel=slack, address=<webhook_url>}` (+ scope: вся страница
  или конкретные компоненты, как у других каналов).
- Доставка уведомлений — POST в этот webhook URL с форматированным сообщением (Slack Block Kit:
  заголовок инцидента, статус, затронутые компоненты, ссылка на страницу).
- Отписка: подписчик удаляет подписку у нас (и/или ревокирует доступ в Slack); храним
  возможность revoke по токену, как для остальных каналов.

**Архитектура.** Доставка через RabbitMQ: routing key `notify.slack.*` → очередь `q.slack` →
`worker-slack` (или общий `worker-webhook`, т.к. механика — это HTTP POST в webhook URL).
**[РЕШЕНО]** На MVP допустимо обслуживать Slack тем же `worker-webhook`, что и исходящие
webhook'и (одинаковый транспорт — HTTP POST), но с отдельной очередью/ключом маршрутизации и
форматированием под Slack Block Kit. Выделять `worker-slack` отдельно — опционально.

**[ТРЕБОВАНИЕ]** Формат сообщений — Slack Block Kit (не plain text), чтобы инциденты выглядели
читаемо (цвет по impact, кнопка-ссылка на страницу).
**[РЕШЕНО для MVP]** Slack App **не** публикуется в App Directory — распространение через прямую
OAuth-ссылку «Add to Slack». Публикация в Directory (с ревью Slack) — опция на будущее, нужна
больше для маркетинга, чем для работы.

## 6. Логика вычисления статусов

- **Общий статус страницы** = производное от текущих статусов всех публичных компонентов:
  - все `operational` → «Все системы работают»;
  - есть `degraded_performance` → «Снижение производительности»;
  - есть `partial_outage` → «Частичная недоступность»;
  - есть `major_outage` → «Крупный сбой»;
  - идут плановые работы (без сбоев) → «Плановые работы».
  Берётся «худший» статус, **НО** `under_maintenance` имеет приоритет показа выше деградации:
  если компонент на плановых работах, на странице показываются именно «Плановые работы», а не
  «деградация», поскольку деградация может быть следствием самих работ.
  **[РЕШЕНО]** Итоговый приоритет отображения (от лучшего к худшему):
  `operational < degraded_performance < partial_outage < major_outage`, при этом активные
  плановые работы (`under_maintenance`) показываются как отдельное состояние «Плановые работы»
  и перекрывают деградацию затронутых ими компонентов.
- **Статус группы** = худший статус среди её компонентов (с тем же правилом для плановых работ).
- **Uptime компонента (90 дней)** = `1 - (суммарное время в не-operational, влияющее на доступность) / период`.
  **[РЕШЕНО]** `under_maintenance` **не** учитывается как downtime — время плановых работ не
  снижает показатель доступности.
- Публичная страница автообновляется на клиенте каждые ~30 секунд (polling или SSE).

## 7. API-контракт (REST, нормативно)

> **Машиночитаемая спецификация:** полный контракт в файле `openapi.yaml` (OpenAPI 3.1,

> Базовый префикс: `/api/v1`. Аутентификация админских/записывающих запросов — заголовок

```

**[РЕШЕНО для MVP] Rate limiting — на уровне инфраструктуры, не в коде приложения.**

под кэш; счётчики rate-limit при необходимости можно добавить позже, но это не требование MVP.
**[РЕШИТЬ]** (некритично) Версионирование публичного API за пределами v1 — отложено.

                         └──────────────┬─────────────┘
                                        │

 │ email    │ │ telegram │ │ webhook  │ │ maintenance  │ │ billing    │

 └──────────┘ └──────────┘ └──────────┘ └──────────────┘ └────────────┘
```
> Каналы Telegram и MAX могут жить в одном воркере мессенджеров или в раздельных
> (`worker-telegram`, `worker-max`) — см. очереди `q.telegram` / `q.max`. `worker-billing`
> инициирует рекуррентные списания и обрабатывает dunning. Отдельно — `worker-import`
> (очередь `q.import`) выполняет длительные задачи миграции со StatusPal/др. (см. 4.3).

**[ТРЕБОВАНИЕ] Принципы:**
- Источник истины — PostgreSQL. Кэш (Redis) — для чтения публичной сводки (rate-limit в MVP на уровне инфраструктуры).
- Все уведомления — **через RabbitMQ** (асинхронно, с ретраями и идемпотентностью по
  `Notification.id`), чтобы пик подписчиков не блокировал API.
- Публичные страницы кэшируются на edge с инвалидацией при изменении статуса/инцидента.
- Кастомные домены: отдельный сервис проверки CNAME и автопродления TLS (ACME).
- Multi-tenant: все запросы изолированы по `status_page_id` (row-level scoping).

### 8.1 Топология RabbitMQ (нормативно)

Очередь сообщений отвечает за доставку уведомлений, исходящие webhook'и и отложенные события.
Выбор RabbitMQ обоснован: фан-аут одного события на тысячи подписчиков по разным каналам,
гарантированная доставка с per-message ack, dead-letter для упавших отправок и отложенные
сообщения для анонсов плановых работ.

**Exchange'и и маршрутизация:**
```
notifications        (topic exchange)
  routing key: notify.<channel>.<event>
    notify.email.incident_update
    notify.telegram.incident_new
    notify.email.maintenance_scheduled
    ...
  ──► queue q.email      (binding: notify.email.*)
  ──► queue q.telegram   (binding: notify.telegram.*)
  ──► queue q.max        (binding: notify.max.*)
  ──► queue q.slack      (binding: notify.slack.*)

webhooks.out         (direct exchange)
  ──► queue q.webhook.out      (исходящие webhook'и: Mattermost/произвольные; Slack-подписка — q.slack)

delayed.events       (x-delayed-message exchange, плагин rabbitmq_delayed_message_exchange)
  ──► публикация с задержкой: напоминание о старте плановых работ,
      авто-переход maintenance scheduled→in_progress по расписанию
```

**Надёжность:**
- **Publisher confirms** на стороне App Service — сообщение считается принятым только
  после ack от брокера.
- **Consumer manual ack** — воркер подтверждает только после успешной доставки
  (письмо отправлено / Telegram принял / webhook вернул 2xx).
- **DLX (dead-letter exchange):** у каждой рабочей очереди настроен `x-dead-letter-exchange`.
  Упавшие после N ретраев сообщения уходят в `q.dlq.<channel>` для разбора и ручной переотправки.
- **Retry с backoff:** через повторную публикацию в delayed exchange с возрастающей задержкой
  (например, 1м → 5м → 30м), счётчик попыток в `Notification.attempts`.
- **Идемпотентность:** `Notification.id` — dedup-ключ; воркер проверяет, не была ли запись
  уже `sent`, перед отправкой (защита от дублей при повторной доставке сообщения брокером).

**Очереди воркеров:**
```
q.email          → worker-email     (SMTP, в т.ч. кастомный SMTP страницы)
q.telegram       → worker-telegram  (Bot API)
q.max            → worker-max       (MAX Bot API; первоклассный канал, см. 4.1)
q.slack          → worker-webhook   (POST в Slack webhook URL, Block Kit; см. 4.4)
q.webhook.out    → worker-webhook   (исходящие интеграции)
q.dlq.*          → ручной разбор / алерт оператору
```

### Закреплённый стек (нормативно)
- **Backend:** **Go (Golang)**. HTTP-роутер — chi или echo (на выбор агента, обосновать в PR).
  Миграции — goose или golang-migrate. БД-доступ — sqlc (типобезопасные запросы) или pgx.
- **Frontend admin:** **React + Feature-Sliced Design (FSD)**. Слои строго по методологии:
  `app / processes / pages / widgets / features / entities / shared`. Бизнес-сущности
  (incident, component, maintenance, subscriber) — в `entities`; пользовательские действия
  (создать инцидент, сменить статус) — в `features`.
- **[РЕШЕНО] Frontend public — отдельный SSR-сервис** (`public-ssr`, например Next.js).
  Публичные страницы статуса рендерятся на сервере для скорости и SEO и **живут независимо от
  админки**: если админка/SPA недоступна, публичная страница статуса продолжает работать
  (это критично — статус-страница должна быть доступнее продукта клиента). Отдельный деплой,
  отдельный образ, читает данные через публичный API (`/summary` и др.).
- **[РЕШЕНО] Маркетинговый лендинг — тоже на SSR** (тот же `public-ssr` или отдельный
  небольшой SSR-инстанс). Нужен для нормальной индексации в поисковиках (SEO): главная, описание,
  тарифы, страницы «перейти с …» для миграции. Без SSR лендинг плохо индексируется.
  Админка (SPA) при этом остаётся client-side — ей SEO не нужно.
- **Очередь сообщений:** **RabbitMQ**. Используется для всех уведомлений (фан-аут на подписчиков),
  входящих/исходящих webhook'ов и отложенных событий. См. раздел 8.1.
- **Кэш:** **Redis** (кэш публичной сводки). Rate-limit в MVP решается на уровне инфраструктуры
  (ingress/API-gateway), не в приложении — см. раздел 7. Redis-счётчики rate-limit — опция на будущее.
- **Telegram / MAX:** воркеры-потребители RabbitMQ; доставка через Bot API соответствующего
  мессенджера. MAX — обязательный канал для рынка РФ (см. 4.1), с учётом лимита ~30 rps.
- **Контейнеризация:** Docker + docker-compose для dev; отдельный образ на каждый сервис
  (api, public-ssr, worker-email, worker-telegram, worker-max, worker-webhook, worker-billing, worker-import).

## 9. Нефункциональные требования

| Категория | Требование |
|-----------|-----------|
| **Доступность** | Публичная страница должна быть доступнее, чем продукт клиента: цель 99.9%, изоляция публичного чтения от админки/воркеров |
| **Производительность** | `/summary` < 200 мс p95 из кэша; рассылка 100k писем без деградации API |
| **Безопасность** | Хэш паролей (argon2/bcrypt); токены хранятся хэшированными; HMAC на webhook; CSRF/CORS политика; rate limiting — на уровне инфраструктуры (ingress/API-gateway), не в коде MVP |
| **Приватность/комплаенс** | Хранение данных в РФ (152-ФЗ); регистрация ПО в реестре отечественного ПО — целевое |
| **Платежи** | Данные карт не хранятся (только токен провайдера, PCI DSS на провайдере); фискализация для MVP через «Мой налог» (режим НПД), приём от физлиц; идемпотентность платежей; верификация подписи webhook'ов провайдера |
| **Идемпотентность** | Повторный webhook/повторная отправка не создаёт дублей (dedup-ключи) |
| **i18n** | RU/EN с первого дня; форматы даты/времени по локали страницы |
| **Наблюдаемость** | Логи, метрики (Prometheus), трейсинг; собственная статус-страница сервиса |
| **Бэкапы** | PITR PostgreSQL; регулярные снапшоты; проверка восстановления |

## 11. План разработки (Roadmap по этапам)

> Каждый этап завершается набором acceptance criteria. Агент не переходит к следующему этапу
> до выполнения текущего.

### Этап 0 — Каркас
- Репозиторий, CI, линтеры, docker-compose (api + postgres + redis + rabbitmq).
- Миграции с пустой схемой; healthcheck-эндпоинт.
- **Приёмка:** `docker compose up` поднимает стек; `/healthz` возвращает 200.

### Этап 1 — Ядро домена (manual-first MVP)
- Сущности: Account, User, StatusPage, ComponentGroup, Component (с `parent_id` — дерево
  подкомпонентов), ComponentStatusHistory.
- Auth (регистрация/логин, сессии/JWT), Membership с ролями.
- CRUD страниц и компонентов (включая вложенные подкомпоненты); ручная смена статуса.
- **Отдельный `public-ssr` сервис:** публичный `/summary` и публичная страница статуса (SSR)
  с общим статусом и деревом компонентов; работает независимо от админки.
- **Маркетинговый лендинг (SSR)** для SEO-индексации (главная, тарифы) — базовая версия.
- **Приёмка:** можно создать страницу, добавить группы/компоненты/подкомпоненты, вручную
  выставить статус, увидеть это на публичной SSR-странице; общий статус считается корректно
  (раздел 6, с приоритетом плановых работ); лендинг отдаётся серверным рендерингом и индексируется.

### Этап 2 — Инциденты и плановые работы
- Incident + IncidentUpdate + IncidentComponent; промежуточные статусы; impact; постмортем.
- Maintenance + MaintenanceUpdate + MaintenanceComponent; статусы работ.
- Авто-производный статус компонентов от активных инцидентов/работ.
- Публичная история инцидентов с фильтрами (компонент, impact) и пагинацией.
- Шаблоны инцидентов.
- **Приёмка:** полный жизненный цикл инцидента investigating→identified→monitoring→resolved
  отражается на публичной странице и в истории; работы scheduled→in_progress→completed
  переводят компоненты в under_maintenance и обратно.

### Этап 3 — Подписки и уведомления
- Subscriber (email с double opt-in, RSS/Atom, iCal).
- Очередь уведомлений на RabbitMQ (topic exchange, DLQ, ретраи с backoff) и идемпотентность; отписка по токену.
- Письма по событиям (новый инцидент, обновление с notify, начало/конец работ).
- Telegram-бот и MAX-бот: подписка на страницу/компоненты, доставка уведомлений
  (учёт лимита ~30 rps у MAX; для MAX нужен верифицированный профиль самозанятого через
  Госуслуги + модерация бота — см. 4.1, организационную часть запускать заранее).
- **Slack-канал подписки** (см. 4.4): Slack App + OAuth «Add to Slack», сохранение
  `Subscriber{channel=slack}`, доставка через `q.slack`/`worker-webhook` в Block Kit-формате.
  Обязателен для запуска.
- **Приёмка:** подписчик получает письмо/сообщение (email, Telegram, MAX, **Slack**) по каждому
  помеченному обновлению; для Slack — через OAuth подключается канал и приходят форматированные
  сообщения; отписка работает; повторная отправка не дублирует.

### Этап 4 — Кастомизация и white-label
- Тема (цвета/макет), тёмная тема, логотип/favicon, формат времени.
- Приватные страницы (пароль/список email), приватные компоненты.
- Собственный домен (CNAME-верификация + ACME TLS), скрытие «Работает на …».
- Custom SMTP / собственный From.
- Виджет статуса (бейдж/сниппет).
- **Приёмка:** страница доступна на кастомном домене по HTTPS; приватная закрыта от анонимов;
  white-label убирает брендинг; виджет встраивается на сторонний сайт.

### Этап 5 — API и интеграции (слой автоматизации)
- ApiToken со scope'ами; полный write-API (раздел 7.2).
- Входящие webhook'и: Grafana, Prometheus, PagerDuty, generic; маппинг на компоненты; HMAC.
- Исходящие webhook'и (Mattermost / произвольный URL). Slack как канал подписки уже реализован
  в этапе 3 через OAuth (см. 4.4); здесь — только технический webhook для прочих систем.
- **Приёмка:** алерт из Grafana создаёт инцидент идемпотентно; внешний скрипт по токену
  открывает/обновляет/закрывает инцидент; повторные webhook'и не плодят дубли.

### Этап 6 — Биллинг и тарифы
> **Рамки MVP (раздел 4.2):** самозанятость + ЮKassa, приём от физлиц, фискализация через НПД.
> Не реализовывать B2B-приём, ОФД и обвязку «на вырост».
- Сущности Subscription, Payment; интерфейс `PaymentProvider` с реализацией под **ЮKassa**.
- Checkout Premium, сохранение платёжного токена, рекуррентные списания (`worker-billing`),
  dunning при неуспехе, откат на Free. Рекурренты предварительно согласовать с менеджером ЮKassa.
- Webhook ЮKassa с верификацией подписи; идемпотентность по `provider_payment_id`.
- Публичная оферта с условиями подключения/отключения автоплатежа.
- Фискализация через «Мой налог» (режим НПД; ЮKassa передаёт сведения в ФНС).
- Единый слой feature-flags: все ограничения тарифов (раздел 10) применяются через флаги.
- **Приёмка:** оплата Premium включает соответствующие фичи; рекуррентное списание проходит
  автоматически; неуспешное списание запускает dunning и в итоге откатывает на Free;
  по каждому платежу формируется чек; повторный webhook не создаёт дублей.

### Этап 7 — Зрелость и миграция
- Расчёт uptime (90 дней) и график; страница релизов (changelog).
- Наблюдаемость (метрики/трейсы), собственная статус-страница сервиса, бэкапы/PITR.
- **Мультиисточниковая миграция** (`worker-import`, раздел 4.3): сущности ImportJob,
  ExternalIdMap; единый интерфейс `Importer` + адаптеры. Порядок реализации адаптеров:
  **StatusPal и Instatus** (первые, основной go-to-market), затем Statuspage и **Статусмейт**
  (best-effort). Импорт: services→компоненты/группы, incidents+updates, maintenances,
  email-подписчиков; маппинг типов/статусов; идемпотентность; отчёт; без рассылки уведомлений.
- **[РЕШИТЬ]** приоритет этапа: миграция — go-to-market рычаг, её можно подтянуть раньше
  (сразу после биллинга), если переманивание клиентов важно на старте.
- **Приёмка:** uptime совпадает с историей состояний; импорт со StatusPal **и** Instatus на
  тестовых аккаунтах переносит структуру сервисов (включая parent/child и private), историю
  инцидентов с хронологией обновлений, плановые работы и email-подписчиков; повторный запуск
  не создаёт дублей; импортированные исторические инциденты не триггерят уведомления; адаптер
  Статусмейта переносит хотя бы компоненты (как best-effort).

## 12. Открытые вопросы (сводка [РЕШИТЬ])

**Решено (зафиксировано в документе):**
1. ✅ Нейминг — **HealthPage / healthpage.ru** (оставлено владельцем; см. раздел 1).
2. ✅ Налоговый статус под биллинг — самозанятость + ЮKassa, приём от физлиц, лимиты НПД как
   рамки MVP; апгрейд на ИП только при подтверждении гипотезы. См. 4.2.
3. ✅ Глубина вложенности компонентов — **подкомпоненты как у Статусмейта** (дерево через
   `parent`). См. 3.2.
4. ✅ Приоритет `under_maintenance` — **плановые работы показываются выше деградации**
   (деградация может быть их следствием); время работ **не** считается downtime в uptime. См. 6.
5. ✅ Rate-limit — **на уровне инфраструктуры** (ingress/API-gateway), не в коде MVP. См. 7.
6. ✅ Публичная часть — **отдельный SSR-сервис** + маркетинговый лендинг на SSR (для SEO). См. 8-стек.
7. ✅ On-prem — **не делаем, только SaaS.**
8. ✅ Ценообразование Premium — **цена Статусмейта −10%** (точную цифру уточнить перед запуском). См. 10.

**Остаётся открытым:**
9. **[ВЕРНУТЬСЯ ПЕРЕД ЗАПУСКОМ МИГРАЦИИ]** 152-ФЗ / согласие подписчиков при импорте. Дефолт —
   повторный opt-in (импортированные email не подтверждены автоматически). Владелец продумывает
   юридическую сторону; исследование и обоснование — в разделе 4.3. Не блокирует разработку
   ядра, блокирует включение импорта подписчиков в проде.
10. MAX-бот: верификация профиля самозанятого через Госуслуги + модерация бота (организационное,
    к этапу 3).
11. Детали миграции (несущественно для первого запуска, второй этап): таблицы маппинга
    кастомных типов StatusPal; размер годовой скидки и длительность триала (дефолт 14 дней);
    нужен ли адаптер Atlassian Statuspage.
