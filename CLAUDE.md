# CLAUDE.md — конституция проекта HealthPage

> Этот файл читается агентом (Claude Code / Codex) **в начале каждой сессии**. Он задаёт
> правила работы. При конфликте инструкций приоритет такой:
> **указание человека в чате > CLAUDE.md > DESIGN.md / openapi.yaml > общие привычки агента.**

---

## 1. Что мы строим

**HealthPage** (`healthpage.ru`) — SaaS-сервис страниц статуса (аналог Статусмейт / Statuspal).
Команды создают публичную страницу статуса своего продукта, **вручную** управляют состоянием
сервисов, публикуют инциденты и плановые работы, а клиенты подписываются на обновления
(email / RSS / iCal / Telegram / MAX / Slack).

**Главный принцип — manual-first.** Статусы и инциденты управляются вручную оператором.
Автоматизация (приём алертов от Grafana/Prometheus/PagerDuty, API) — опциональный слой поверх
ручного ядра, не замена. Не превращай продукт в probe-based мониторинг.

Текущая цель — **MVP для проверки спроса**, не масштабирование. Не закладывай инфраструктуру и
сложность «на вырост» сверх того, что явно описано в DESIGN.md.

## 2. Источники истины (куда обращаться)

| Вопрос | Файл | Статус |
|--------|------|--------|
| Архитектура, модель данных, фичи, логика | `./DESIGN.md` | Нормативный |
| API-контракт (эндпоинты, схемы, enum'ы) | `./openapi.yaml` | Нормативный, провалиден |
| Декомпозиция на задачи, порядок работ | `./ROADMAP.md` | Рабочий |
| Что сделано, текущее состояние, решения | `./MEMORY.md` | Обновляется каждую сессию |
| Как запустить/развернуть: порты, env, секреты, команды | `./INFRASTRUCTURE.md` | Операционный, наполняется по ходу |
| Правила работы (этот файл) | `./CLAUDE.md` | Нормативный |

**Правила обращения:**
- Любой код должен соответствовать `DESIGN.md` и `openapi.yaml`. Если код требует отклонения —
  **остановись и спроси человека** (см. раздел 4), не меняй контракт молча.
- Enum'ы, имена статусов, структура API — **нормативны**. Не переименовывай.
- Если в `DESIGN.md` нашёл `[РЕШИТЬ]` по теме, которую сейчас трогаешь, — это развилка для
  человека, не решай сам (см. раздел 4).
- `[ВЕРНУТЬСЯ ПЕРЕД ЗАПУСКОМ ...]` — жёсткий стоп-маркер: фича не уходит в прод, пока человек
  не снимет его.
- **Архитектура vs инфраструктура:** логическая архитектура (какие сервисы, как связаны) — в
  `DESIGN.md §8`; факт развёртывания (порты, env, секреты, команды запуска) — в
  `INFRASTRUCTURE.md`. Не дублировать диаграмму сервисов в INFRASTRUCTURE.md. Секреты — только
  имена переменных и `.env`; **никогда не записывать значения секретов в файлы или git.**

## 3. Технический стек (закреплён, не меняй без согласования)

- **Backend:** Go. Роутер chi или echo. БД-доступ — sqlc или pgx. Миграции — goose или
  golang-migrate. Layout — стандартный Go (`cmd/`, `internal/`).
- **Frontend admin:** React + **Feature-Sliced Design** (слои `app/processes/pages/widgets/
  features/entities/shared` строго). SPA, client-side.
- **Frontend public:** Next.js (SSR) — публичные страницы статуса **и** лендинг в одном сервисе
  `public-ssr` (для MVP; заложить возможность вынести лендинг отдельно позже).
- **Очередь:** RabbitMQ (все уведомления, webhook'и, отложенные события). Топология — DESIGN §8.1.
- **Кэш:** Redis (кэш публичной сводки). Rate-limit в MVP — на уровне инфраструктуры, не в коде.
- **БД:** PostgreSQL (источник истины).
- **Контейнеризация:** Docker + docker-compose (dev), отдельный образ на сервис.
- **Платежи:** ЮKassa за интерфейсом `PaymentProvider`.

## 4. Разделение ответственности (КРИТИЧНО)

**Человек (владелец) решает и делает:**
- **Все git-операции: коммиты и мержи — ТОЛЬКО человек.** Агент НЕ коммитит и НЕ мержит.
- Все архитектурные решения и продуктовые развилки (схема БД на новом уровне, выбор библиотек
  из вариантов, любые `[РЕШИТЬ]`, изменения контракта/модели данных).
- Снятие стоп-маркеров `[ВЕРНУТЬСЯ ПЕРЕД ЗАПУСКОМ ...]`.

**Агент решает сам (мелкие решения):**
- Имена внутренних переменных/функций, структура файлов внутри слоя, приватные хелперы.
- Реализация в рамках уже принятого контракта и модели данных.
- Выбор между эквивалентными мелочами, не влияющими на API/схему/архитектуру.

**Когда агент обязан остановиться и спросить человека:**
- Нужно изменить `openapi.yaml`, модель данных или enum.
- Встретил `[РЕШИТЬ]` по текущей теме.
- Нужно выбрать библиотеку/подход, где DESIGN.md даёт варианты («chi или echo»).
- Задача требует решения, которого нет в источниках истины.
- Что-то в DESIGN.md противоречит openapi.yaml или самому себе.

Формат остановки: коротко описать развилку, варианты, свою рекомендацию — и ждать ответа.
Не угадывай за человека по важным вопросам.

## 5. Git-процесс

- **Агент НИКОГДА не выполняет `git commit`, `git push`, `git merge`.** Это исключительно
  прерогатива человека.
- Агент готовит изменения в рабочем дереве, описывает что сделал, и предлагает текст коммита
  (conventional commits: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`).
- Человек ревьюит, коммитит и мержит сам.
- Перед тем как сказать «готово», агент убеждается, что код собирается и тесты проходят
  (см. раздел 6) — чтобы человек ревьюил рабочий код, а не сломанный.
- Не трогай `.git/`, не меняй git-историю, не создавай теги.

## 6. Качество кода (что агент проверяет ДО «готово»)

- **Сборка:** `go build ./...` (бэк), сборка фронтов — без ошибок.
- **Тесты:** покрывать доменную логику юнит-тестами; для хендлеров — тесты на контракт.
  Запускать `go test ./...` перед сдачей. Не сдавать с падающими тестами.
- **Линтеры:** `gofmt`/`goimports`, `golangci-lint` (бэк); eslint/prettier (фронт).
- **Идемпотентность и безопасность** — по DESIGN §9: пароли хэшируются (argon2/bcrypt), токены
  хранятся хэшированными, webhook'и проверяются по подписи, карты не хранятся.
- **i18n:** публичные тексты — RU/EN с первого дня.
- Не оставляй TODO без отметки в MEMORY.md.

## 7. Организация репозитория (монорепо)

```
healthpage/
├── CLAUDE.md  MEMORY.md  ROADMAP.md  DESIGN.md  openapi.yaml
├── docker-compose.yml        # postgres, redis, rabbitmq для dev
├── go.work                   # Go workspace
├── backend/
│   ├── cmd/                  # точки входа: api, worker-email, worker-telegram,
│   │                         #   worker-max, worker-webhook, worker-billing, worker-import
│   ├── internal/
│   │   ├── domain/           # сущности: incident, component, maintenance, subscriber, ...
│   │   ├── api/              # HTTP-хендлеры (по openapi.yaml)
│   │   ├── store/            # доступ к БД (sqlc/pgx)
│   │   ├── queue/            # RabbitMQ publish/consume
│   │   └── ...
│   ├── migrations/           # goose/golang-migrate
│   └── go.mod
├── frontend/
│   ├── admin/                # React + FSD (SPA-админка)
│   └── public-ssr/           # Next.js: публичные страницы статуса + лендинг
└── shared/
    └── api-types/            # типы, сгенерированные из openapi.yaml (TS + Go)
```

**Правила:**
- Контракт один — `openapi.yaml`. Типы для фронта и бэка **генерируются** из него, не пишутся
  руками. Изменение API едет одним коммитом с потребителями (плюс монорепо).
- Доменная логика — только в `backend/internal/`, наружу не экспортируется.
- FSD на фронте соблюдать строго: бизнес-сущности в `entities`, действия в `features`.
- Лендинг и статус-страницы пока в одном `public-ssr`; код держать так, чтобы лендинг можно
  было вынести в отдельный сервис без переписывания.

## 8. Рабочий цикл сессии (агент следует каждый раз)

1. **Прочитать** `MEMORY.md` (где мы остановились) и `ROADMAP.md` (что дальше).
2. Свериться с `DESIGN.md` / `openapi.yaml` по текущей задаче.
3. Выполнить **одну задачу из ROADMAP** (или часть, если крупная). Не хвататься за несколько
   сразу — последовательная работа, один агент.
4. Проверить качество (раздел 6): сборка, тесты, линтеры.
5. **Обновить `MEMORY.md`**: что сделано, какие решения приняты, что осталось, новые TODO.
6. Отметить прогресс в `ROADMAP.md` (задача → done / in progress).
7. Предложить текст коммита. **Дальше git — за человеком.**

Если упёрся в развилку (раздел 4) — остановись на шаге 3, опиши, жди человека.

---

## 9. Чего НЕ делать

- Не коммитить, не мержить, не пушить.
- Не менять `openapi.yaml`, модель данных, enum'ы без согласования.
- Не закладывать сложность «на вырост» сверх DESIGN.md (это MVP для проверки спроса).
- Не реализовывать probe-based мониторинг (ломает manual-first философию).
- Не хранить данные карт; не хранить секреты/токены в коде или в git.
- Не игнорировать стоп-маркеры `[ВЕРНУТЬСЯ ПЕРЕД ЗАПУСКОМ ...]`.
- Не реализовывать приём платежей от юрлиц/ИП (MVP — самозанятость, только физлица; DESIGN §4.2).
- Не рассылать уведомления по историческим/импортированным данным.
